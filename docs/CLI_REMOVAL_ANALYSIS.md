## 🔍 详细差距分析

### 1. 方案管理功能（核心差距）

**CLI 优势**：
```python
# 创建方案 - 完整的交互流程
- 选择房间类型（下拉菜单）
- 选择楼层（动态查询可用楼层）
- 输入座位号
- 设置时间参数（带默认值提示）
- 即时校验（座位是否存在）
- 立即保存到 data/plans.yaml
```

**GUI 现状**：
- 只有占位按钮
- 点击后弹出"功能即将上线"提示
- 用户必须切换到 CLI 或手动编辑 YAML

**影响**：⚠️ **严重** - 这是用户首次使用的必经流程

### 2. 批量修改时间

**CLI 优势**：
```python
# 多选方案 + 批量更新
- 支持多选（复选框）
- 可选择性更新（留空保持原值）
- 一次操作修改多个方案
```

**GUI 现状**：完全缺失

**影响**：⚠️ **中等** - 不常用但很方便，缺失降低用户体验

### 3. 删除方案

**CLI 优势**：
- 多选删除
- 确认提示

**GUI 现状**：完全缺失

**影响**：⚠️ **中等** - 用户需要手动编辑 YAML 文件

### 4. 浏览房间与座位

**CLI 优势**：
```python
# 查看所有可用座位
[自习室]
  - A区1楼 (ID 123): [A001, A002, A003, ... (+50)]
  - A区2楼 (ID 124): [A201, A202, ...]
[阅览室]
  - B区1楼 (ID 456): [B001, B002, ...]
```

**GUI 现状**：完全缺失

**影响**：⚠️ **中等** - 用户不知道有哪些座位可选，创建方案时需要瞎猜

### 5. 定时预约参数覆盖

**CLI 优势**：
```python
# 每次执行可临时调整
max_trials = input_float("最大重试次数 [5]", 5)
retry_delay = input_float("重试延迟 [1.0]", 1.0)
window_wait_seconds = ...
window_poll_interval = ...
```

**GUI 现状**：
- 只能使用 config.yaml 的默认值
- 无法在执行前临时调整

**影响**：⚠️ **较轻** - 大多数情况用默认值即可，高级用户可编辑配置文件

## 📈 代码依赖分析

### CLI 模块结构
```
cli/
├── __init__.py         # 空
├── app.py             # InteractiveApp 主类（~350 行）
├── menu.py            # 菜单选择工具（select_menu, multi_select_menu）
├── prompts.py         # 输入提示工具（input_int, parse_execute_time 等）
└── views.py           # 格式化输出（print_plan_table, format_progress_line）
```

### 依赖关系
```
main.py
  └─→ cli/app.py (InteractiveApp)
        ├─→ cli/menu.py (交互菜单)
        ├─→ cli/prompts.py (输入工具)
        ├─→ cli/views.py (格式化输出)
        └─→ services/* (与 GUI 共享)

gui/main_window.py
  ├─→ cli/prompts.py (复用 parse_execute_time) ⚠️
  └─→ services/* (与 CLI 共享)
```

**关键发现**：GUI 目前依赖 `cli/prompts.py` 的 `parse_execute_time()` 函数！

## 🎯 移除 CLI 的影响评估

### 直接影响

1. **用户无法创建方案** ⛔
   - 首次使用障碍
   - 必须手动编辑 YAML（技术门槛高）

2. **用户无法管理方案** ⛔
   - 无法删除错误方案
   - 无法批量修改时间
   - 必须手动编辑 YAML

3. **用户无法浏览座位** ⚠️
   - 不知道有哪些座位
   - 创建方案时需要猜测座位号

4. **GUI 代码依赖断裂** ⛔
   - `gui/main_window.py` 导入 `cli/prompts.py`
   - 移除 CLI 后需要重构

### 间接影响

5. **文档需要大幅更新** 📝
   - README.md 中多处提到 CLI
   - 快速开始指南基于 CLI 创建方案
   - 需要重写所有用户指引

6. **降低灵活性** ⚠️
   - 某些场景 CLI 更高效（SSH、脚本化）
   - 失去"轻量级命令行工具"的定位

7. **开发测试不便** 🔧
   - CLI 便于快速测试功能
   - GUI 启动慢，调试不便

## 💡 移除可行性结论

### ❌ **当前不可行** 

**原因**：
1. ⛔ GUI 功能覆盖率仅 44%，核心方案管理功能缺失
2. ⛔ 用户首次使用即遇到障碍（无法创建方案）
3. ⛔ 存在代码依赖（gui → cli/prompts.py）
4. ⚠️ 需要大量额外开发工作（至少 5 个功能）

## 🔄 替代方案

### 方案 A：保留 CLI（推荐）✅

**理由**：
- CLI 和 GUI 各有优势，互补而非替代
- CLI 适合：SSH 环境、脚本化、快速测试
- GUI 适合：日常使用、可视化、新手友好
- 代码量不大（~1200 行），维护成本低

**实现**：
- 保持现状
- 继续完善 GUI 功能
- 文档明确两者定位

### 方案 B：分阶段迁移（可选）⚠️

**Phase 1**: 补齐 GUI 功能（2-4 周）
- 实现 P0 清单（方案创建、删除）
- 解除依赖

**Phase 2**: 增强 GUI（1-2 周）  
- 实现 P1 清单（批量修改、房间浏览）

**Phase 3**: 移除 CLI（1 天）
- 删除 cli/ 目录
- 更新文档
- 发布新版本

**Phase 4**: 后续维护
- 用户反馈
- 补充遗漏功能

### 方案 C：精简 CLI（折中）🤔

**保留部分 CLI**：
- 仅保留方案管理功能（create, modify, delete）
- 作为 GUI 的补充工具
- 移除重复的抢座功能

**实现**：
```python
# main.py
if "--manage" in sys.argv:
    # 进入简化版 CLI（仅方案管理）
    PlanManagerCLI().run()
elif "--gui" in sys.argv:
    # GUI 模式
    run_gui()
else:
    # 默认：直接 GUI
    run_gui()
```

## 🎯 最终建议

### 推荐方案：**保留 CLI**（方案 A）

**理由**：
1. ✅ CLI 和 GUI 定位不同，可以并存
2. ✅ 维护成本低（代码稳定，基本不需要改动）
3. ✅ 提供灵活性（SSH、脚本、快速测试）
4. ✅ 用户可以自由选择
5. ✅ 无需额外开发工作

**定位**：
- **CLI**: 管理工具（方案管理、座位浏览）+ 快速测试
- **GUI**: 日常使用（抢座、监控、新手友好）
- **--run-now**: 自动化任务（无人值守）

### 如果坚持移除 CLI

**前置条件**：
1. 完成上述 P0 清单（必须）
2. 完成上述 P1 清单（强烈建议）
3. 工作量：15-24 小时
4. 时间周期：2-4 周

**风险**：
- ⚠️ 开发周期长
- ⚠️ 可能引入新 bug
- ⚠️ 用户学习成本（习惯 CLI 的用户需要适应）
- ⚠️ 失去命令行工具的优势
